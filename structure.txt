MeetJot - Project Structure

Goal
- Electron desktop app (React renderer) + local Python FastAPI sidecar
- Focus on: UI shell, local API bridge, database, integrations, approval workflow
- AI/STT/LLM modules exist in-repo but are not part of current development scope

Repository layout (proposed)
/
  README.md
  COOKBOOK.md
  structure.txt

Scope notes
- We keep AI/models-related files in the structure so another dev can implement them.
- For current development, we focus on: frontend UI, backend API/services, integrations, and SQLite.
- AI modules should be treated as placeholders/stubs until implemented by the AI dev.

Data flow (non-AI)
- Renderer shows a "Review Deck" of draft actions stored in SQLite.
- Renderer calls backend over HTTP (localhost) to:
  - list drafts
  - edit drafts
  - approve/reject drafts
  - on approve: backend executes integration call (JIRA/GCal) and persists status

SQLite (baseline)
- actions
  - id (uuid)
  - status (PENDING|APPROVED|REJECTED|ERROR)
  - tool_type (JIRA_TICKET|G_CAL_EVENT)
  - draft_payload (json)
  - final_payload (json)
  - created_at (timestamp)

Local paths (Windows)
- App data: %APPDATA%/MeetJot/
- DB file: %APPDATA%/MeetJot/MeetJot.db

Dev ergonomics
- Dev mode starts:
  - Vite renderer
  - Electron main
  - Python FastAPI on localhost:8000
- Prod build:
  - Electron builder bundles backend executable (Python frozen separately)

Notes
- This structure assumes TypeScript for Electron/React and standard FastAPI for Python.
- We implement the staging/review/execution system now; AI modules are listed above but not implemented in this phase.
